# Horus 全球合伙人计划：机制增强与容错加固实施计划（2026-03-04）

## 1. 目标
针对全球合伙人机制在极限行情、大并发结算以及自动化流转方面存在的长尾缺陷，进行专项加固。确保该顶级机制在面对极限对抗时具备强防脆弱性。

## 2. 识别到的隐患点与加固项

### 2.1 隐患一：末位清退的“并列拉踩断档”（Tie-breaker缺失）
- **现状**：当前淘汰逻辑仅基于某一核心指标（如当月净入金）进行排名截断，当处于第 90 名至第 95 名的合伙人指标完全一致时，系统排序可能产生随机性（导致同分的人命运不同）。
- **加固目标**：引入多维度 Tie-breaker（加时赛/决胜局）算法。
  - **初筛**：当月推荐激活的有效 managed 资产规模（如果首要指标是净入金）。
  - **决胜 1**：历史累计参与总金额。
  - **决胜 2**：绑定资金账户的系统年龄（先到先得/留存优势）。
- **交付物**：重构 `web/lib/participation-program/partner-program.ts` 中的排序算法，增加多栏位排序，并在 Dry-Run 接口中输出排序依据明细。

### 2.2 隐患二：空缺席位的价格抢跑（缺乏队列与防秒杀保护）
- **现状**：席位腾出并被超管改价后，外部看到即可直接 API 抢拍。在牛市中极易被脚本党/工作室用发包机秒杀。
- **加固目标**：空单席位置为竞拍/防秒杀模式。
  - **策略**：当有空缺席位时，进入“申购队列”模式或“白名单抽签制”。
  - **交付物**：新增 `/api/partners/seats/apply` 接口，结合 Redis rate-limit 与 CAPTCHA 拦截脚本；或在后台增加“开启申购期 -> 收集意向 -> 随机开奖”的业务状态机。

### 2.3 隐患三：退款 SLA 仅有告警无自动化打款
- **现状**：超时 7 天不退款仅抛出告警（Watchdog），实际退款仍依赖人工处理。
- **加固目标**：引入半自动化/全自动化资金释放链路。
  - **策略**：利用现有的管理资金流向引擎，在触发 7 天 SLA 时，将对应的 USDT 额度从平台预留池（ReserveFund）自动过账划拨给淘汰者；或如果安全级别不够，至少生成批量放款待签名的 tx payload 供财务一键 multisig。
- **交付物**：开发 `auto-refund-executor` 脚本，联动 `ReserveFundLedger`。

### 2.4 隐患四：自动化调度的基建断带（人工单点依赖）
- **现状**：淘汰脚本需要通过 cron 或人工点击触发，存在进程假死或单点故障风险。
- **加固目标**：调度任务的高可用保障与重试。
  - **策略**：将淘汰任务重构为带有状态机的持久化 Task (`PartnerEliminationTask`)，在数据库层记录进行到了排名的哪一步，即使进程中断，下一次拉起时能断点续传。
- **交付物**：新增任务状态模型与幂等执行器（类似于目前的 ManagedLiquidationTask）。
